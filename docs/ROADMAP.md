# Roadmap Astra Nexus

## Текущее состояние

Проект находится на этапе локального MVP.

Уже есть:
- базовый Python-проект;
- локальное окружение;
- NoDriver / ChatGPT Web provider;
- безопасный browser profile;
- команды диагностики, clean, login, dom-probe, smoke, ask;
- тесты для NoDriver lifecycle;
- базовая документация по NoDriver.

Главная цель MVP — не автономный Codex и не self-improving система, а личная AI-команда, которой можно дать сложную задачу один раз, а она разложит её между несколькими ролями, проверит, улучшит и вернёт отполированный результат.

## Итерация 2 — AI Team Orchestration

Цель: сделать слой команды, где разные агенты имеют разные роли, задачи, стиль мышления и порядок работы.

### Базовые роли

- Coordinator — принимает задачу, понимает, что вообще нужно сделать.
- Analyst — разбирает факты, структуру, вводные данные.
- Critic — ищет ошибки, слабые места, противоречия и недосказанность.
- Editor — улучшает текст, структуру, стиль и понятность.
- QA Controller — проверяет, выполнена ли задача и не потеряны ли требования.
- Final Composer — собирает финальный ответ в нормальный вид.

### Нужно сделать

- Добавить профили агентов.
- Добавить системные инструкции для каждого агента.
- Добавить pipeline выполнения:
  - последовательный;
  - частично параллельный;
  - с возможностью повторной проверки.
- Добавить режим “обычный диалог” и режим “задача”.
- Сделать так, чтобы система не воспринимала каждое сообщение пользователя как задачу.
- Добавить intent-router:
  - обычный разговор;
  - новая задача;
  - уточнение к текущей задаче;
  - файл для анализа;
  - остановка;
  - продолжение.
- Добавить логику передачи результата одного агента другому.
- Добавить внутренние события:
  - агент начал думать;
  - агент задал уточнение;
  - агент передал результат другому;
  - агент нашёл проблему;
  - агент завершил работу.

### Критерий готовности

Итерация считается готовой, если можно отправить сложную текстовую задачу, получить несколько внутренних ответов агентов, пройти через критику/редактуру/финальную сборку и получить один итоговый результат.

### Добавлено в текущем скелете

- Выделен доменный слой `astra_nexus.team`.
- Добавлены роли `coordinator`, `analyst`, `critic`, `editor`, `qa_controller`, `final_composer`.
- Добавлены domain-сущности `TeamRun`, `AgentProfile`, `AgentTask`, `AgentResult`, `RunEvent`.
- Добавлен последовательный in-memory pipeline без Telegram, NoDriver и реальной параллельности.
- Добавлен `TeamProvider` contract и `FakeTeamProvider` для unit-тестов.
- Добавлены события `run_started`, `run_finished`, `run_failed`, `agent_started`, `agent_finished`, `agent_failed`.
- Добавлен внутренний event stream и human-readable сообщения агентов для будущего Telegram/UI bridge.
- Добавлен Team Intake / Intent Router, чтобы обычный диалог не запускал AI-команду автоматически.
- Добавлен Team Runtime / Conversation Controller для связки intake, orchestrator, workspace, resume и message sink.
- Добавлен Telegram Team Bridge v1 как тонкая aiogram-оболочка над runtime/controller.
- Добавлен Telegram Team Jobs v1: background jobs, `/status`, `/stopall` и защита от дублей в одном чате.
- Добавлен Files for Team Tasks v1: attachments manifest, `input_files/`, txt/md extraction и file context в prompts.
- Добавлен Team Dialogue v1: живой transcript агентов, `team_chat.json`/`team_chat.md` и разделение main-chat/log-chat.
- Добавлен Parallel Agents Foundation v1: execution plan, parallel fake-provider режим, fallback на sequential для NoDriver.
- Добавлен Team Review Protocol v1: task brief, критерии качества, review notes, revision requests, QA decision, один revision loop и final package.
- Добавлен Telegram Live Team v1: aiogram polling runtime, allowlist чатов, файлы, `/status`, `/stopall`, main/log chat split, typing и safe fake-provider default.
- Добавлен Telegram Team Atmosphere v1: humanized main-chat rendering, agent voice styles,
  message budget, technical log chat и безопасные NoDriver window defaults.
- Добавлен Persistent Team Run Registry v1: файловый индекс `TEAM_RUNS_DIR`, `/runs`,
  `/status` после restart-like состояния и tolerant чтение повреждённых `run.json`.
- Добавлен Attachments Extraction v2: извлечение текста из `txt`, `md`, `docx`, `pdf`,
  prompt truncation, расширенная metadata attachments и безопасные ошибки extraction.
- Добавлен Output Artifacts v1: итоговые markdown/json артефакты run, индекс
  `artifacts/index.md`, manifest, Telegram notice и отправка `final_answer.md`/`index.md`.

## Итерация 3 — Telegram task flow

Цель: сделать Telegram основным интерфейсом управления командой.

### Нужно сделать

- Подключить Telegram bot polling. (v1 добавлен как bridge над `TeamConversationController`.)
- Сделать основной чат пользователя. (v1 отправляет `user_visible_reply` и main-chat сообщения.)
- Сделать отдельный лог-чат или супергруппу для отображения работы команды. (v1 поддерживает `TEAM_TELEGRAM_LOG_CHAT_ID`.)
- Добавить background jobs, чтобы Telegram handler не ждал долгий pipeline. (v1 добавлен через `TeamJobManager`.)
- Добавить отображение:
  - кто из агентов сейчас думает;
  - кто что предложил;
  - кто кому передал задачу;
  - где возникла ошибка;
  - какой итоговый результат готов.
- Сделать человеческие сообщения агентов, не сухие системные статусы.
- Добавить foundation для живого командного диалога агентов. (v1 добавлен как dialogue transcript и main-chat stream.)
- Добавить команду `/stopall`.
- Добавить аварийную отмену всех активных runs.
- Сделать нормальное поведение при обычном разговоре:
  - агент может просто ответить;
  - не надо всегда запускать весь pipeline;
  - сложный pipeline включается только когда это действительно задача.

### Критерий готовности

Итерация считается готовой, если пользователь может написать задачу в Telegram, увидеть “командную работу” агентов в лог-чате и получить финальный ответ в основном чате.

### Добавлено в текущем скелете

- `astra-nexus-team-telegram-bot` запускает aiogram 3 polling runtime поверх
  `TeamConversationController`, `TeamJobManager`, `TelegramTeamBridge`, attachments и
  intake-router.
- Default provider для Telegram runtime - `fake`; `NoDriverTeamProvider` импортируется и
  стартует только при `TEAM_TELEGRAM_PROVIDER=nodriver`.
- Поддержаны text/caption задачи, `/status`, `/stopall`, Telegram document/photo input
  attachments и безопасное поведение для файла без инструкции.
- Main chat получает человеческие реплики агентов и финал; log chat получает технические
  run/job события только если задан `TEAM_TELEGRAM_LOG_CHAT_ID`.
- Добавлен `astra-nexus-team-telegram-live-preview` для dry-run симуляции casual text, task,
  status, stopall и файлов без Telegram API.

## Итерация 4 — NoDriver / ChatGPT Web provider hardening

Цель: довести NoDriver provider до стабильного транспортного слоя.

### Уже сделано

- browser profile;
- clean;
- diagnose;
- login;
- dom-probe;
- smoke;
- ask;
- retry после Failed to connect;
- защита от ложного profile_locked;
- тесты lifecycle.

### Нужно доделать

- Добавить стресс-тесты на серию ask-запросов.
- Добавить понятный healthcheck provider.
- Добавить graceful fallback при временной ошибке ChatGPT.
- Добавить более чистые user-facing ошибки.
- Добавить настройки задержек, retry и timeout.
- Не ломать профиль и cookies.
- Не удалять пользовательскую сессию при clean.

### Критерий готовности

Итерация считается готовой, если 10–20 подряд `ask` проходят без ручного вмешательства или корректно восстанавливаются через retry.

## Итерация 5 — Файлы задач и артефакты

Цель: дать системе возможность работать не только с текстом, но и с файлами.

### Нужно сделать

- Принимать файлы из Telegram.
- Сохранять их в workspace run.
- Поддержать входные форматы:
  - txt;
  - md;
  - docx;
  - pdf.
- Научиться извлекать текст из файлов.
- Передавать содержимое файлов нужным агентам.
- Делать итоговые артефакты:
  - Markdown;
  - DOCX;
  - PDF.
- Отправлять готовые файлы обратно в Telegram.
- Сохранять связи:
  - какой файл пришёл;
  - кто его анализировал;
  - какой результат получился;
  - какой артефакт создан.

### Критерий готовности

Итерация считается готовой, если пользователь может отправить файл, агент-критик его проверит, редактор улучшит, финальный агент соберёт результат, а Telegram вернёт готовый документ.

## Итерация 6 — Параллельные агенты и отдельные ChatGPT-сессии

Цель: сделать работу команды ближе к “нескольким живым специалистам”, а не к одному prompt-chain.

### Нужно сделать

- Добавить поддержку нескольких agent sessions.
- Добавить foundation execution plan для будущей параллельности. (v1 добавлен: fake-provider parallel, NoDriver sequential fallback.)
- Продумать, когда агент работает:
  - через общий provider;
  - через отдельный ChatGPT-чат;
  - через отдельный профиль/контекст.
- Сделать очередь задач.
- Сделать параллельное выполнение независимых этапов.
- Добавить сбор результатов от нескольких агентов.
- Добавить арбитра/координатора, который решает, чьё мнение важнее.
- Добавить защиту от путаницы ролей.
- Не давать агентам бесконтрольно запускать друг друга.
- Не делать self-improving режим в MVP.

### Критерий готовности

Итерация считается готовой, если несколько агентов могут параллельно обработать одну задачу, после чего координатор соберёт их выводы и отдаст нормальный финальный результат.

- Добавить project memory.
- Добавить user preference memory.
- Разделить:
  - факты;
  - стиль;
  - прошлые задачи;
  - рабочие правила;
  - запреты.
- Добавить короткие personality-инструкции.
- Добавить ограничения, чтобы характер не мешал качеству.
- Сделать агентов живыми, но не клоунскими.
- Добавить human-like сообщения в Telegram:
  - короткие реплики;
  - реакции;
  - уточнения;
  - передачу задач друг другу.

### Критерий готовности

Итерация считается готовой, если агенты стабильно сохраняют роли, стиль, память и не превращают обычный разговор в формальную задачу.
