# Roadmap Astra Nexus

## 1. Цель проекта

Astra Nexus — локальная AI-команда, которая работает через Telegram и ChatGPT Web.

Главная идея: пользователь даёт одну задачу, а система сама понимает, что с ней делать: ответить быстро, запустить команду агентов, разобрать файл, создать документ, дождаться долгого reasoning-ответа, скачать готовый файл из ChatGPT Web и вернуть результат в Telegram.

Цель MVP — не автономный Codex, не self-improving система и не полноценный агент-разработчик. Цель MVP — удобная личная AI-команда, которая помогает думать, проверять, писать, анализировать файлы и собирать качественный результат.

---

## 2. Базовые принципы

Система работает локально и управляется пользователем.

Telegram — основной интерфейс управления.

ChatGPT Web — основной интеллектуальный provider и основной генератор файлов.

NoDriver — транспортный слой для работы с ChatGPT Web, а не центр архитектуры.

Обычный разговор не должен запускать тяжёлый pipeline.

Система сама должна определять режим работы:
- обычный быстрый ответ;
- стандартная командная задача;
- глубокий командный разбор;
- задача с файлом;
- задача на создание файла;
- debug/diagnostic режим.

Пользователь не должен вручную выбирать режим в большинстве случаев.

Каждый агент должен иметь свою роль, стиль мышления, характер и отдельный контекст.

Основной Telegram-чат должен получать только понятные человеческие сообщения и финальный результат.

Технические события, ошибки, run_id, workspace, debug и диагностика должны уходить в log chat.

Если пользователь просит файл, Astra должна стараться получить настоящий файл из ChatGPT Web, а не подменять это локальным костылём.

Служебные файлы вроде `final_answer.md` и `index.md` не должны отправляться пользователю без запроса.

## 3. Текущее состояние

Проект находится на стадии локального live MVP.

Уже есть:

- Python-проект и локальное окружение;
- настройки через `.env`;
- NoDriver / ChatGPT Web provider;
- безопасный browser profile;
- команды `clean`, `doctor`, `login`, `dom-probe`, `smoke`, `ask`;
- диагностика NoDriver;
- восстановление после части проблем запуска Chrome;
- Telegram bot runtime на aiogram;
- основной Telegram-чат и log chat;
- allowlist Telegram-чатов;
- команды `/help`, `/health`, `/status`, `/runs`, `/stopall`;
- background jobs для задач;
- workspace для каждого run;
- registry последних runs;
- роли агентов:
  - coordinator;
  - analyst;
  - critic;
  - editor;
  - qa_controller;
  - final_composer;
- последовательный team pipeline;
- fake provider для тестов;
- NoDriver provider для реальных задач;
- intake router, чтобы отличать обычный диалог от задачи;
- работа с файлами `txt`, `md`, `docx`, `pdf`;
- сохранение входных файлов в workspace;
- извлечение текста из файлов;
- базовые artifacts;
- базовая атмосфера AI-команды;
- HTML-render финального ответа из ChatGPT Web в Telegram;
- тесты, smoke-проверки и документация.

Что пока не доведено до целевого вида:

- агенты ещё не живут в отдельных ChatGPT-чатах;
- реальная параллельность агентов ещё не готова;
- часть реплик агентов всё ещё шаблонная;
- выбор GPT-5.5 Thinking / extended reasoning ещё нужно сделать надёжным;
- долгие reasoning-задачи нужно лучше сопровождать статусами;
- скачивание файлов, созданных ChatGPT Web, ещё не реализовано как основной путь;
- `telegram_bridge.py` и `chatgpt_client.py` разрастаются и требуют разделения;
- dashboard / mini app откладывается в ультрадальний этап.

## 4. Ближайший приоритет

Сейчас главный фокус — довести live MVP до нормального ежедневного состояния.

Порядок ближайших работ:

1. стабилизировать текущий Telegram + NoDriver live flow;
2. починить настоящий download файлов из ChatGPT Web;
3. сделать отдельные ChatGPT-чаты для агентов;
4. добавить автоматическое определение режима задачи;
5. добавить автоматический execution plan вместо жёсткой цепочки;
6. подготовить реальную параллельность агентов;
7. разрезать крупные файлы `telegram_bridge.py` и `chatgpt_client.py`;
8. отполировать UX Telegram: меньше шаблонов, больше нормальных сообщений.

## 5. Этап 1 — Stabilize Live MVP

Цель: сделать так, чтобы текущая система стабильно выполняла задачу из Telegram через NoDriver и возвращала финальный результат.

Нужно сделать:

- проверить полный сценарий Telegram → NoDriver → agents → final answer;
- убедиться, что `/status` показывает реальный `run_id`, workspace и текущего агента;
- убедиться, что `/stopall` корректно отменяет долгую задачу;
- убрать дубли сообщений в main chat;
- убрать технический шум из main chat;
- проверить, что log chat получает все технические события;
- проверить, что финальный ответ всегда доходит в Telegram;
- проверить длинные ответы и корректный chunking;
- проверить HTML/Telegram formatting;
- добавить нормальную диагностику, если ChatGPT Web завис на thought-only или пустом turn;
- закрепить рабочие `.env` настройки для local live mode.

Критерий готовности:

Пользователь отправляет в Telegram задачу, система проходит pipeline, возвращает финальный ответ, а в log chat остаётся понятный технический след.

## 6. Этап 2 — ChatGPT Web Artifact Downloader

Цель: научить Astra Nexus скачивать настоящие файлы, созданные ChatGPT Web.

Это один из главных этапов, потому что ChatGPT Web сам умеет создавать файлы. Astra не должна в основном сценарии подменять это локальной генерацией `requested_output.md`.

Если пользователь просит файл, основной сценарий такой:

1. Astra определяет, что нужен файл:
   - `md`;
   - `txt`;
   - `docx`;
   - `xlsx`;
   - `pptx`;
   - `pdf`;
   - image;
   - zip.

2. Astra отправляет задачу в ChatGPT Web с явным требованием создать скачиваемый файл.

3. NoDriver ищет в текущем assistant turn:
   - file card;
   - download button;
   - attachment;
   - filename chip;
   - ссылку или кнопку скачивания.

4. Astra скачивает реальный файл из ChatGPT Web UI.

5. Файл сохраняется в workspace run:
   - `requested_files/`;
   - `requested_file_request.json`;
   - `requested_file_download_result.json`;
   - debug artifact detector output.

6. Telegram отправляет пользователю именно скачанный файл.

7. Если ChatGPT Web написал “готово, файл можно скачать”, но реального файла нет, это не считается успехом. Нужно сделать retry или вернуть понятную ошибку.

Нужно сделать:

- artifact detector для текущего assistant turn;
- download manager для Chrome/NoDriver;
- отдельный downloads dir для run;
- проверку расширения, размера и завершения скачивания;
- retry, если ChatGPT написал про файл, но файл не прикрепил;
- отправку только requested file в Telegram;
- debug dump HTML/JSON при неудаче;
- CLI:
  - `astra-nexus-nodriver-dump-artifacts`;
  - `astra-nexus-nodriver-artifact-test`;
  - `astra-nexus-team-requested-file-preview`.

Критерий готовности:

Пользователь просит Word/Excel/Markdown/PDF/презентацию, ChatGPT Web создаёт файл, Astra скачивает его и отправляет в Telegram без служебных `final_answer.md` и `index.md`.

Цель: уйти от ощущения, что один ChatGPT каждый раз притворяется разными ролями, и перейти к постоянным агентским чатам.

Нужно сделать:

- добавить `AgentChatSessionRegistry`;
- хранить для каждого агента отдельный ChatGPT conversation URL;
- добавить команды инициализации агентских чатов;
- один раз прописывать каждому агенту:
  - роль;
  - характер;
  - стиль ответа;
  - правила;
  - ограничения;
  - формат результата;
- сделать режим, где orchestrator переключается между готовыми agent chats;
- добавить fallback: если agent chat потерян, создать новый;
- сохранять связь:
  - `agent_id`;
  - `chat_url`;
  - `created_at`;
  - `last_used_at`;
  - `status`;
- добавить диагностику активных agent sessions;
- не включать настоящую параллельность на этом этапе.

Критерий готовности:

Каждый агент работает в своём ChatGPT-чате, не путает роль, не получает заново огромный role prompt каждый раз и сохраняет ощущение постоянного специалиста.

## 8. Этап 4 — Automatic Task Modes

Цель: сделать так, чтобы Astra сама понимала, какой режим нужен для конкретного сообщения.

Пользователь не должен вручную выбирать режим. Система должна определять это автоматически.

Режимы:

- `simple_answer` — обычный быстрый ответ без команды;
- `team_standard` — стандартная командная работа;
- `team_deep` — глубокий разбор с критикой, QA и возможной доработкой;
- `file_task` — анализ или обработка входного файла;
- `file_generation` — создание выходного файла через ChatGPT Web;
- `debug_mode` — максимум логов и диагностики;
- `casual_chat` — обычный разговор без запуска run.

Нужно сделать:

- улучшить intent router;
- добавить определение сложности задачи;
- добавить определение необходимости файла;
- добавить определение необходимости глубокого reasoning;
- добавить определение, нужен ли полный team pipeline;
- не запускать тяжёлую команду на обычный вопрос;
- сохранять выбранный режим в `run.json`;
- показывать режим в `/status` и log chat.

Критерий готовности:

Пользователь пишет обычный вопрос — Astra отвечает просто. Пользователь пишет сложную задачу — Astra сама запускает нужный командный режим. Пользователь просит файл — Astra сама включает file generation flow.

## 9. Этап 5 — Automatic Execution Plan

Цель: заменить жёсткую цепочку агентов на автоматически построенный план выполнения.

Сейчас базовая цепочка выглядит так:

`coordinator → analyst → critic → editor → qa_controller → final_composer`

Это нормально для MVP, но не для всех задач.

Нужно сделать:

- coordinator строит `execution_plan`;
- plan содержит:
  - шаги;
  - нужных агентов;
  - зависимости;
  - parallel/sequential режим;
  - критерии готовности;
  - нужен ли QA;
  - нужен ли final composer;
- простые задачи получают короткий plan;
- сложные задачи получают глубокий plan;
- файловые задачи получают file-aware plan;
- независимые этапы могут выполняться параллельно;
- результаты собираются в deterministic order;
- plan сохраняется в workspace;
- plan отображается в log chat.

Критерий готовности:

Astra не гоняет всех агентов всегда, а автоматически строит подходящий план под задачу.

## 10. Этап 6 — Model and Reasoning Control

Цель: надёжно выбирать нужную модель ChatGPT и режим расширенного мышления.

Нужно сделать:

- добавить настройку preferred model;
- добавить настройку preferred reasoning mode;
- добавить проверку текущей выбранной модели в ChatGPT UI;
- добавить диагностику, какая модель реально активна;
- если включено `require preferred model`, не запускать задачу при неправильной модели;
- отдельно настроить:
  - быстрый режим;
  - reasoning mode;
  - deep/extended thinking;
- добавить понятную ошибку, если модель не найдена или UI изменился;
- не делать вид, что включено extended thinking, если оно реально не включено.

Критерий готовности:

Перед запуском задачи система понимает, какая модель выбрана, какой reasoning mode активен, и может остановиться, если режим не соответствует настройкам.

## 11. Этап 7 — Long Running Tasks

Цель: нормально поддержать задачи, где ChatGPT может думать 10–20 минут и дольше.

Нужно сделать:

- убрать жёсткий timeout для live Telegram задач;
- оставить bounded timeout только для smoke/diagnostic команд;
- отличать:
  - ChatGPT думает;
  - ChatGPT пишет intermediate answer;
  - ChatGPT пишет final answer;
  - ChatGPT завис;
  - UI idle, но final не найден;
- показывать progress в log chat;
- в main chat писать только редкие и полезные статусы;
- сделать `/status` информативным во время долгого run;
- сделать `/stopall` безопасной отменой;
- сохранять partial state в workspace;
- добавить resume с последнего успешного агента.

Критерий готовности:

Долгая reasoning-задача может выполняться без ложного timeout, пользователь видит состояние через `/status`, а при сбое можно понять, где именно всё остановилось.

## 12. Этап 8 — Stop / Cancel v2

Цель: сделать отмену задач аккуратной и предсказуемой.

Это не самый главный этап, но он нужен для долгих задач и параллельной работы.

Нужно сделать:

- `/stopall` отменяет active job;
- если ChatGPT сейчас генерирует, попытаться нажать stop generating;
- корректно освободить browser session;
- снять profile lock;
- сохранить cancelled state;
- не оставлять Chrome думать в фоне;
- записать cancellation event в log chat;
- сохранить partial workspace;
- не повреждать browser profile и cookies.

Критерий готовности:

После `/stopall` задача реально останавливается, профиль освобождается, `/status` показывает cancelled, следующий запуск не упирается в lock.

## 13. Этап 9 — Human Team Dialogue v2

Цель: сделать Telegram-переписку похожей на работу живой команды, а не на набор шаблонных статусов.

Нужно сделать:

- перестать отправлять шаблонные фразы как основной UX;
- брать короткие осмысленные реплики из реального ответа агента;
- отправлять main-chat сообщения только когда они реально полезны;
- ограничить количество сообщений в main chat;
- оставить технические события только в log chat;
- добавить dedupe одинаковых реплик;
- сделать стили агентов живыми, но не клоунскими;
- добавить реакции/короткие передачи между агентами, если это не мешает задаче;
- сделать режимы атмосферы:
  - `silent`;
  - `minimal`;
  - `normal`;
  - `verbose`;
- по умолчанию оставить `minimal` или `normal`.

Критерий готовности:

В основном Telegram-чате видно, что команда работает, но без мусора, повторов и искусственных фраз. Реплики агентов связаны с реальной задачей.

## 15. Этап 11 — Parallel Agents v1

Цель: сделать реальную параллельную работу агентов, но без хаоса и profile lock.

Параллельность должна строиться на Agent Chat Sessions и отдельных browser profiles / sessions.

Нужно сделать:

- добавить несколько browser profiles;
- привязать profile к agent session;
- разрешить параллельность только для независимых этапов;
- оставить coordinator и final_composer как контрольные точки;
- добавить лимит одновременных агентов;
- добавить очередь задач;
- добавить timeout/stop/resume для каждого agent task;
- добавить сбор результатов в deterministic order;
- не запускать несколько агентов через один и тот же Chrome profile;
- не делать self-improving режим.

Базовая схема:

- coordinator работает первым;
- analyst и critic могут работать параллельно;
- editor ждёт analyst и critic;
- QA ждёт editor;
- final_composer ждёт QA.

Критерий готовности:

Несколько агентов реально работают параллельно в разных ChatGPT-сессиях или профилях, результаты корректно собираются, роли не смешиваются.

## 16. Этап 12 — Orchestrator / Executor Split

Цель: разделить “кто думает” и “кто выполняет реальные действия”.

Orchestrator не должен быть одновременно Telegram-ботом, NoDriver-парсером, скачивателем файлов и автором ответа.

Orchestrator отвечает за:

- постановку задачи;
- план;
- роли агентов;
- порядок выполнения;
- статусы;
- события;
- передачу результатов;
- финальную сборку.

Executors отвечают за реальные действия:

- `NoDriverProvider`;
- `TelegramSender`;
- `ChatGPTWebArtifactDownloader`;
- `WorkspaceWriter`;
- `AttachmentExtractor`;
- будущие file/doc/pptx/xlsx/image executors.

Нужно сделать:

- выделить общий executor interface;
- убрать прямые действия из orchestrator;
- не смешивать Telegram sending с team logic;
- не смешивать artifact downloading с ChatGPT response parsing;
- не смешивать workspace saving с agent reasoning;
- сделать зависимости явными.

Критерий готовности:

В коде видно разделение: team layer думает и управляет, executor layer выполняет конкретные действия.

## 20. Этап 16 — ChatGPT Account Memory / Context Strategy

Цель: использовать память и контекст без отдельной тяжёлой memory-системы в MVP.

Отдельную локальную memory-БД пока не делаем.

Что используем вместо этого:

- память самого ChatGPT-аккаунта;
- постоянные agent chat sessions;
- историю Telegram-чата как возможный источник контекста;
- workspace прошлых runs только как reference/debug;
- ручные project notes в будущем, если понадобится.

Что нужно исследовать позже:

- можно ли аккуратно подмешивать историю Telegram;
- нужен ли parser Telegram export;
- можно ли давать агентам краткий context pack из прошлых задач;
- как не превратить память в кашу;
- где граница между полезным контекстом и лишним шумом.

Критерий готовности для MVP:

Агенты используют постоянные ChatGPT-чаты и память аккаунта ChatGPT. Отдельная локальная память не обязательна.

## 23. Не входит в MVP

Пока не делаем:

- self-improving систему;
- бесконтрольное изменение собственного кода агентами;
- полноценный Codex replacement;
- автономное выполнение shell-команд агентами без подтверждения;
- продакшн-деплой;
- облачную многопользовательскую версию;
- сложные права доступа;
- marketplace агентов;
- отдельную тяжёлую локальную memory-систему;
- quality score для каждого run;
- source zip exporter;
- полноценный dashboard до стабилизации основной системы;
- полностью автоматическое исправление UI ChatGPT при любом изменении сайта.

## 24. Ближайшая задача для Codex

ChatGPT Web Artifact Downloader.

Минимальный результат:

- тесты проходят;
- старый text answer flow не ломается.
